# slide_smith v1.2 — Exemplar-first (Reference Deck) Design

**Status:** proposed (targeting v1.2)

## Summary

v1.2 introduces an *exemplar-first* workflow: given **(A)** a Markdown document (content) and **(B)** an example PPTX whose *style should be followed* (the **reference deck**), slide_smith generates a new PPTX that **only uses layouts/placeholders/styles proven to exist** in the reference.

slide_smith remains **LLM-free**. All behavior is **deterministic Python**: parsing, layout selection, compilation, rendering, and validation.

## Goals

- **Reference deck as style source**: treat an example PPTX as the primary source of truth for theme/layout/style.
- **Layout-driven rendering**: prefer “instantiate known layouts + fill placeholders” over arbitrary freeform shapes.
- **Stable intermediate representation**: separate *content planning* (Markdown → SlidePlan) from *render mechanics* (SlidePlan → DeckSpec → PPTX).
- **Enforceable style**: provide a validator to check that outputs comply with the reference deck’s affordances.

## Non-goals (v1.2)

- No LLM or ML (no generative/semantic rewriting).
- No invention of new layouts or styles not present in reference.
- No full-fidelity “clone everything” of reference (animations, complex chart theming) beyond what we can deterministically validate.

---

## Key Concepts

### Reference Deck
A PPTX provided by the user that represents the desired style.

### StyleProfile
A compact, machine-readable summary of what slide_smith is allowed to do safely with the reference deck.

- Captures **theme tokens** (fonts/colors when detectable), **available layouts**, **placeholder schemas**, and optional inferred **margins/grid hints**.
- Intended to be small enough for other agents/tools to reason about without reading the whole PPTX.

### SlidePlan
A deterministic intermediate representation derived from Markdown: slide intents and semantic fields.

### DeckSpec
A renderable, low-level plan that binds SlidePlan to a specific reference deck’s layouts and placeholders.

---

## CLI / UX (proposed)

A minimal pipeline that mirrors the internal architecture:

1) Analyze reference

```bash
slide-smith analyze --reference ref.pptx --out style.profile.json
```

2) Plan from Markdown

```bash
slide-smith plan --input deck.md --out slide.plan.json
```

3) Render from plan using reference

```bash
slide-smith render --reference ref.pptx --plan slide.plan.json --out out.pptx
```

4) Validate output

```bash
slide-smith validate --reference ref.pptx --pptx out.pptx
```

Optional convenience:

- `slide-smith compile --reference ref.pptx --plan slide.plan.json --out deck.spec.json`
- `slide-smith list-layouts --reference ref.pptx` (with placeholder schema + thumbnails if feasible)

## Data Models

### StyleProfile (JSON)

> Stored as `style.profile.json`.

```json
{
  "version": "1",
  "reference": {
    "path": "ref.pptx",
    "sha256": "...",
    "slideSize": {"widthEmu": 12192000, "heightEmu": 6858000}
  },
  "theme": {
    "fonts": {
      "major": "Aptos Display",
      "minor": "Aptos"
    },
    "colors": {
      "accent1": "#4472C4",
      "accent2": "#ED7D31"
    }
  },
  "layouts": [
    {
      "layoutId": "layout:TitleSlide",
      "name": "Title Slide",
      "placeholders": [
        {"ph": "title", "idx": 0, "type": "title", "bbox": {"x": 123, "y": 456, "w": 789, "h": 101}},
        {"ph": "subtitle", "idx": 1, "type": "body", "bbox": {"x": 123, "y": 900, "w": 789, "h": 202}}
      ]
    }
  ],
  "constraints": {
    "placeholderOnly": true,
    "allowNewShapes": false
  },
  "hints": {
    "margins": {"left": 0.6, "right": 0.6, "top": 0.4, "bottom": 0.4},
    "grid": {"columns": 12, "gutter": 0.2}
  }
}
```

Notes:
- Theme extraction in PPTX can be partial; fields may be missing.
- Layout IDs must be stable across runs for the same reference (derive from layout XML part name + placeholder schema hash).
- `bbox` is in a consistent unit (EMU preferred) and always relative to slide.

### SlidePlan (JSON)

> Stored as `slide.plan.json`.

```json
{
  "version": "1",
  "source": {"path": "deck.md", "sha256": "..."},
  "slides": [
    {
      "intent": "title",
      "fields": {
        "title": "My Deck",
        "subtitle": "Optional subtitle"
      },
      "notes": "Speaker notes if present"
    },
    {
      "intent": "bullets",
      "fields": {
        "title": "Agenda",
        "bullets": ["One", "Two", "Three"]
      }
    },
    {
      "intent": "two_column",
      "fields": {
        "title": "Approach",
        "leftBullets": ["A", "B"],
        "rightImage": {"path": "assets/diagram.png", "alt": "Diagram"}
      }
    }
  ]
}
```

Deterministic Markdown rules (v1.2):
- `# H1` starts a new slide (title intent if first).
- `## H2` starts a new slide (section intent).
- Bullet lists under a heading become `bullets` intent.
- Images become `image` intent or are attached to the nearest slide, depending on context.

### DeckSpec (JSON)

> Stored as `deck.spec.json` (optional intermediate output).

```json
{
  "version": "1",
  "reference": {"path": "ref.pptx", "sha256": "..."},
  "slides": [
    {
      "layoutId": "layout:TitleSlide",
      "fills": [
        {"placeholder": {"type": "title", "idx": 0}, "text": "My Deck"},
        {"placeholder": {"type": "body", "idx": 1}, "text": "Optional subtitle"}
      ]
    }
  ]
}
```

Constraints:
- v1.2 default path must not create arbitrary shapes.
- Image placement uses picture placeholders when available; otherwise hard error (unless an escape hatch is explicitly enabled by flag).

## Layout Selection (deterministic)

The compiler selects a layout for each SlidePlan slide by matching placeholder schemas.

### Placeholder schema matching
Represent each layout by a normalized signature:

- Counts and types: `title(1) body(1) picture(0) ...`
- Optional role hints (e.g. left/right body placeholders) inferred by x-order of `bbox`.

### Heuristic rules (v1.2)

- `intent=title` → layout with `title>=1` and optional `subtitle/body>=1`.
- `intent=section` → layout with `title>=1` and minimal other placeholders.
- `intent=bullets` → layout with `title>=1` and `body>=1`.
- `intent=two_column`:
  - prefer `title>=1` + (`body>=2`) for left/right columns, OR
  - `title>=1` + (`body>=1` + `picture>=1`).

Ties are broken deterministically:
1) Prefer layouts seen most frequently in reference.
2) Prefer layouts with tighter match (fewer unused placeholders).
3) Stable lexical order of `layoutId`.

If no layout matches: emit an actionable error listing candidate layouts and missing placeholders.

---

## Rendering (deterministic)

Renderer instantiates slides as:

- Create new slide using chosen `layoutId` from the reference deck.
- Fill placeholders only:
  - text: set placeholder text frame contents
  - images: replace picture placeholder content

No geometry changes by default (keep the reference-defined bbox).

---

## Validation

`slide-smith validate` checks:

- **Layout compliance**: each slide uses a known layout (or is an allowed blank).
- **Placeholder-only edits**: no unexpected shapes inserted (unless allowed by config/flag).
- **Theme compliance (best-effort)**:
  - fonts/colors are within theme tokens when detectable
  - warn (not fail) if theme extraction incomplete

Output:
- Human-readable summary + JSON report (`validate.report.json`).

## Release Plan (v1.2)

### Milestones

0. Docs + schemas (0.5–1 day)
- Write this doc
- Add JSON schema stubs under `schemas/`

1. Analyze reference → StyleProfile (2–4 days)
- Layout + placeholder extraction
- Stable layout IDs
- Snapshot tests on 1–2 reference decks

2. Markdown → SlidePlan (2–3 days)
- Deterministic parser
- Snapshot tests

3. Compile SlidePlan → DeckSpec (2–4 days)
- Deterministic layout selection
- Clear error messages

4. Render DeckSpec → PPTX (2–4 days)
- Placeholder fill (text + images)
- Asset handling

5. Validate output (1–3 days)
- Layout + placeholder-only enforcement
- Theme checks (best-effort)

6. CLI polish + docs + release notes (1 day)

### Acceptance Criteria

- Given a reference deck with standard title/bullets/two-column layouts, slide_smith can generate a new deck from Markdown using only those layouts.
- Running validate on the output reports **0 errors** for the placeholder-only path.
- Behavior is deterministic across runs (same inputs → same outputs).
