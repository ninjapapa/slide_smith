# python-pptx Feasibility Notes

## Bottom line

`python-pptx` looks feasible for the MVP of `slide_smith`.

It is strong at **programmatically creating and editing PowerPoint decks** from Python, especially when the workflow is based on:

- opening a presentation or template
- adding slides from known layouts
- filling placeholders
- adding text, images, tables, and charts
- saving a `.pptx` result

That maps well to the first version of an agent-first deck generator.

## What the library clearly supports

From the official docs, `python-pptx` supports these core patterns well:

### Presentation-level operations
- create a new presentation with `Presentation()`
- open an existing `.pptx`
- save to a new `.pptx`
- inspect slide masters and slide layouts
- set slide size and presentation properties

### Slide-level operations
- add a slide from a selected layout via `slides.add_slide(...)`
- access slides by index
- inspect slide names, backgrounds, and notes state

### Placeholder-driven authoring
This is one of the strongest parts of the library for our use case.

The docs emphasize using placeholders as stable insertion points. A slide created from a layout can expose title/body/picture placeholders, and those placeholders can be addressed by placeholder index (`idx`).

That is good for agent workflows because it gives us:
- a stable contract between template and content
- a simpler way to map markdown/JSON into slides
- less need for manual x/y positioning in the MVP

### Shapes and content insertion
The docs show support for adding and editing:
- text frames and paragraphs
- text boxes
- pictures
- tables
- charts
- connectors
- group shapes
- movies
- OLE objects

For `slide_smith`, the practical high-value subset is:
- text
- pictures
- tables
- charts

That is already enough for a strong MVP.

## What this means for the MVP

The following MVP goals appear very feasible:

### 1. Create deck from markdown or JSON
Feasible.

A simple pipeline could be:
- parse markdown/JSON into a deck spec
- choose a slide layout per slide type
- write title/body/image/table/chart content into placeholders or known shapes
- save the deck

### 2. Apply a known template
Feasible.

A good pattern is likely:
- use a real `.pptx` file as the template source
- open it with `Presentation(template_path)`
- add slides using that file’s layouts
- target placeholders instead of freehand positioning when possible

This fits the agent-first model well because templates can be made explicit and inspectable.

### 3. Add slides during iterative editing
Feasible.

Adding slides is a first-class library operation.

### 4. Modify text and replace content on existing slides
Feasible, especially when the target slides are built from known layouts/placeholders.

This suggests an early design principle:

**Decks generated by `slide_smith` should carry enough structure and conventions that later edits remain predictable.**

## Feasibility concerns / rough edges

This is where the library gets less clean.

### 1. Slide deletion is not clearly a first-class operation
The docs clearly expose `slides.add_slide(...)`, but they do **not** appear to expose an equally clear high-level `delete slide` API on the `Slides` collection.

That matters because deleting slides is part of the desired workflow.

Implication:
- **adding slides is easy**
- **deleting slides may require lower-level XML/package manipulation or a controlled workaround**

For MVP planning, this means slide deletion should be treated as **possible but risky/awkward** until proven in code.

### 2. Slide reordering and duplication may also be awkward
The docs present clean indexed access to slides, but not an obvious first-class reorder/duplicate workflow.

Implication:
- generate in the right order where possible
- avoid promising sophisticated deck surgery too early
- keep edit operations narrow at first

### 3. Template extraction from arbitrary decks is much harder than template usage
Using a known `.pptx` template is feasible.

But your bigger ambition — taking an example deck from a user and summarizing it into a reusable template representation — is a harder problem.

The library can inspect presentation structure, layouts, shapes, placeholders, and text, but **inferring design intent** from an arbitrary deck is a separate layer of logic that we would need to build.

Examples of hard questions:
- which slides are the same slide type?
- which text box is a title vs. just decorative text?
- which visual properties are essential template rules vs. one-off formatting?
- how should charts/tables/placeholders be abstracted into reusable slots?

So:
- **template consumption** = feasible for MVP
- **template learning/extraction** = feasible later, but not an MVP feature

### 4. Precise visual fidelity can get painful
If we rely on raw coordinates and arbitrary shapes, the system can become fragile quickly.

For example:
- manual textbox placement
- dynamic overflow handling
- auto-layout decisions
- fitting variable-length content cleanly

This suggests we should prefer:
- curated layouts
- placeholders
- a small set of slide archetypes
- constrained content rules

rather than trying to support arbitrary visual composition immediately.

## Recommended product implications

### Recommendation 1: build around template layouts and placeholders
This seems like the best path.

Instead of saying “agents can create any slide however they want,” say:
- here are the supported slide types
- here are the template layouts
- here are the allowed content fields

That is much more robust.

### Recommendation 2: make the internal deck spec structured
A JSON-like deck model will help a lot.

For example:

```json
{
  "title": "Q2 Business Review",
  "slides": [
    {
      "type": "title_and_bullets",
      "title": "Highlights",
      "bullets": ["Revenue up 20%", "Margin improved", "Pipeline strong"]
    },
    {
      "type": "image_left_text_right",
      "title": "Product demo",
      "body": "New workflow reduces setup time.",
      "image": "demo.png"
    }
  ]
}
```

That gives the agent a stable contract and reduces ambiguity.

### Recommendation 3: keep edit commands narrow in MVP
For MVP, prefer operations like:
- create deck
- add slide
- update slide title/body
- replace image

Be cautious about claiming:
- arbitrary slide deletion
- arbitrary slide reordering
- arbitrary deep modification of imported decks

### Recommendation 4: treat imported example decks as references first, templates second
A practical roadmap could be:

1. **Phase 1**: use hand-curated templates
2. **Phase 2**: inspect imported decks and summarize structure
3. **Phase 3**: infer reusable template models from imported decks

That sequence reduces risk.

## Suggested MVP feasibility verdict

### High confidence
- create presentation from scratch
- create from markdown/JSON via intermediate deck spec
- use `.pptx` templates
- fill placeholders
- add text/images/tables/charts
- save result

### Medium confidence
- edit previously generated decks reliably
- inspect template structure and expose it to agents
- support limited “add slide after X” workflows

### Low confidence / likely awkward without lower-level work
- delete arbitrary slide cleanly
- reorder slides cleanly
- duplicate slides cleanly
- infer reusable template semantics from arbitrary user decks automatically
- maintain perfect visual fidelity across all imported decks

## Recommendation for `slide_smith`

`python-pptx` is a good foundation **if we design the application around its strengths**.

That means:
- template-driven generation
- placeholder-first authoring
- structured deck specs
- simple, reliable edit operations
- narrow MVP scope

It is probably the right library for the first version.

But the product should avoid pretending that arbitrary PowerPoint understanding and arbitrary deck surgery are easy. Those are likely higher-layer problems beyond the base library.
